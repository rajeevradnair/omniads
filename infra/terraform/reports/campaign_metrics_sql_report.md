# Campaign Metrics SQL Report

OmniAds now includes a campaign analytics layer that turns raw ad-serving events into operator-facing business metrics. The real-time decision engine answers the question, “Which ads should be returned right now?” The analytics layer answers a different question: “Did the ad platform perform well after those decisions were made?” This matters because an ad platform is not complete if it can only return VAST responses. Operators, advertisers, and platform engineers also need to know whether campaigns delivered, whether creatives received impressions and clicks, whether ad pods filled correctly, and whether no-fill problems occurred.

The reports in this release are built on top of the OmniAds event lake. Events are written as partitioned JSONL files locally and then uploaded to S3 using the same partition layout. The local layout is `data/synthetic_events/event_type=<event_type>/year=<yyyy>/month=<mm>/day=<dd>/events.jsonl`. The AWS layout is `s3://<event-lake-bucket>/events/event_type=<event_type>/year=<yyyy>/month=<mm>/day=<dd>/events.jsonl`. This structure allows both local DuckDB and AWS Athena to query the same logical event lake using partition columns such as `event_type`, `year`, `month`, and `day`.

The event types used by these reports are `ad_decision_created`, `vast_returned`, `impression_received`, `click_received`, `conversion_received`, and `no_fill_returned`. An `ad_decision_created` event means the ADS Gateway created an ad decision attempt. A `vast_returned` event means the VAST Service returned an ad response. An `impression_received` event means a tracking endpoint recorded that an ad impression occurred. A `click_received` event means a viewer clicked an ad. A `conversion_received` event represents a downstream success event. A `no_fill_returned` event means the platform could not return an ad for a request.

The local analytics path uses DuckDB to query the partitioned JSONL event files directly. The command is `python scripts/run_local_analytics_reports.py`. This command creates an in-memory `omniads_events` view over `data/synthetic_events/event_type=*/year=*/month=*/day=*/events.jsonl` and runs the local SQL reports under `sql/local/`. The local reports are `fill_rate.sql`, `campaign_performance.sql`, `creative_performance.sql`, `no_fill_reasons.sql`, and `pacing_health.sql`. Local validation is useful because it lets us confirm the report logic before relying on AWS Athena.

The AWS analytics path uses Amazon S3 for the event lake, AWS Glue Data Catalog for table metadata, and Amazon Athena for SQL execution. The external Athena table is `omniads_analytics.omniads_events`. The Athena SQL reports live under `sql/athena/`, and Terraform stores them as Athena saved queries using `infra/terraform/athena_named_queries.tf`. This makes the analytics layer repeatable and infrastructure-managed instead of relying on manually pasted console queries.

The first report is `fill_rate.sql`. It answers the question, “Did the platform return ads for ad decision attempts?” The formula is `fill_rate = vast_returned / ad_decision_created`. It also calculates `no_fill_rate = no_fill_returned / ad_decision_created`. A high fill rate means OmniAds is successfully returning ads. A low fill rate would tell an operator to investigate campaign availability, targeting, frequency caps, budget pacing, ad pod packing, or VAST rendering.

The second report is `campaign_performance.sql`. It answers the question, “Which campaigns are getting impressions, clicks, and conversions?” It groups events by `campaign_id` and calculates impressions, clicks, conversions, CTR, and CVR. The formulas are `CTR = click_received / impression_received` and `CVR = conversion_received / click_received`. This report is advertiser-facing because it shows whether campaigns are producing measurable engagement.

The third report is `creative_performance.sql`. It answers the question, “Which exact ad creatives are performing best?” It groups outcomes by `campaign_id` and `creative_id`. This matters because two creatives inside the same campaign can perform differently. A campaign may be healthy overall while one specific creative is weak. Creative-level reporting helps operators identify which video ads should be promoted, paused, replaced, or tested further.

The fourth report is `no_fill_reasons.sql`. It answers the question, “Why did OmniAds fail to return an ad?” The report groups `no_fill_returned` events by `attributes.no_fill_reason`, falling back to `status` or `UNKNOWN` when needed. Future no-fill reasons may include `NO_ACTIVE_CAMPAIGNS`, `NO_TARGETING_MATCH`, `FREQUENCY_CAP_EXHAUSTED`, `BUDGET_EXHAUSTED`, and `NO_ADS_FIT_POD`. This report is important for revenue recovery and system diagnosis because a raw no-fill count is not actionable unless the operator knows what caused it.

The fifth report is `pacing_health.sql`. It answers the question, “Is the decision engine producing enough viable candidates to fill ad breaks?” It reads decision-funnel metadata from the `attributes` JSON on `ad_decision_created` events, including `pod_fill_rate`, `pacing_allowed_count`, and `ranked_candidate_count`. This report is different from campaign performance. Campaign performance measures viewer engagement, while pacing and pod health measure whether the ad decision engine had enough valid ads to choose from and whether the pod packer filled the ad break properly.

The elements in this analytics layer are the partitioned event lake, local validation before AWS execution, S3-backed analytics path, Glue external table, Athena SQL reports, Terraform-managed Athena saved queries, version-controlled SQL logic, and clear operator-facing metric definitions. These are all patterns used in real analytics systems: raw events are stored, metadata is cataloged, SQL reports are versioned, and saved queries are deployed through infrastructure-as-code.

The elements are that the events are synthetic, impression and click events may be manually triggered, conversion events may not exist yet, no-fill events may not exist yet, and the reports are not yet connected to a dashboard. The event pipeline is also still synchronous/local rather than fully asynchronous through Kinesis or Firehose. Budget and pacing logic are simplified at this stage.

For validation, `fill_rate.sql` should return rows if `ad_decision_created` and `vast_returned` events exist. `pacing_health.sql` should return rows if `ad_decision_created` events include decision attributes such as `pod_fill_rate`, `pacing_allowed_count`, and `ranked_candidate_count`. `campaign_performance.sql` and `creative_performance.sql` may be empty unless impression and click tracking events have been generated. `no_fill_reasons.sql` may be empty unless no-fill events have been generated. Empty results are acceptable when the relevant event type does not exist; query failures are not acceptable.

In summary, this release adds an analytics layer that makes OmniAds observable from a business and operator perspective. Local DuckDB reports prove the SQL logic against partitioned JSONL files, while AWS Athena reports prove the same analytics pattern over the S3 event lake. The reports calculate fill rate, no-fill rate, campaign performance, creative performance, CTR, CVR, and pacing health. Terraform-managed Athena saved queries make the reporting layer repeatable, versioned, and AWS-operationalized.
